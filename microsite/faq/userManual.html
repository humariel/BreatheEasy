User Manual

In this section we'll explain how you use CityZoom.

Description & main features

 On this section, we explain some of the main features of the system, as well as an explanation on its usage.

 CityZoom is a platform designed to provide the city manager a web application to gather environmental information from a city and act accordingly. A mobile application was also developed for citizens that allows them to give more environmental data of the place where they are.

 The application has 2 main interfaces with different goals:

  • The homepage dashboard, designed to give an overview of the environmental data where the user is;
  • The map view, designed to help the city manager check the city overall status with heatmaps over the different streams of data;

 There are many types of sensor verticals and streams allowed on the system. You can add sensors to the platform if the given sensor has the specified vertical(s).

 Besides the access to all environmental data, you can also add alerts to specific locations and streams, with a threshold value.

 The mobile application, destinated to the citizens, allows them to give some data from their smartphones. In the future it would give them an access-token in order to create devices and post data to the CityZoom web application.

 The homepage dashboard has the following main features:
  • View current location area devices & streams;
  • View current municipality devices & streams:
  • View current alarms;
  • View triggered alarms;

 The map view has the following main features:
  • Check all sensors' streams values in real-time;
  • Check the location of all the sensors in a map in real-time;
  • Heatmap vision of every stream in hexagon and parish;
  • View data snapshots from previous moments;

 For the sensor technician, the platform has the following features:
  • API to add sensors dynamically (CRUD compliant);
  • API to add values dynamically;
  • API to add alerts dynamically (CRUD compliant);
  • Mobile sensors support.


How to access it

 In this section, we explain how to have access to the different IoT City applications.


Web portal and admin portal

 For accessing the web portal and admin portal you just need to go to this link. When logged, you will be redirected to the dashboard web portal page. You will need an account, there's an option to do that in the login page.


Mobile application

 The mobile application is designed to run on android operating system (version 4.0 or higher). You need to download the .apk file and install it on your smartphone. The apps are free to use, and no login is required.


How it works

 In this section, we present the homepage dashboard, the map view admin portal and the mobile application along with an explanation of usage.


Homepage Dashboard

 The first view is the authentication page. If the user is not authenticated, it doesn’t have the permission to see any information gathered. If the username and password entered by the user match with a valid account, the webpage will redirect the user to the main dashboard page. Each account has devices and alerts associated. Devices have certain sets of streams, depending on the verticals supported.


Authentication view

 The main page of the web portal contains a dashboard with information gathered by the sensors. For each type of metric, there is a card with information about the last values retrieved by the sensors.
 Bellow the cards, there is a notification panel. The first tab contains a list of all the triggered alerts, alongside with its information. An alert can be added by the user on the sensor metrics page or alerts page. An alert is a maximum or minimum threshold for a number of subscriptions, that can be turned on or off according to a schedule. It is also possible to act when an alarm is triggered by adding a actuator (e.g. when it is 7 p.m., turn on public lighting), sending a message to all the actuators.

 The "Triggered alerts" tab shows the occurrences by date. These can have two states: active and inactive. If an triggered alert is active, it means that the sensor values are currently crossing a threshold. When it stops, the state will be inactive. If the user wants to dismiss the alert notification, he can do it and it will no longer be in the "Triggered alerts".

 On the left sidebar, we can access different pages of the web portal.


Main dashboard page

 The map view contains a map with the location of all the sensors of the user, with a filter to select the streams that are visible on the map. When the user clicks on top of a sensor, it will appear a popup box with information about that sensor and the streams of that sensor .


Map with some sensors for that user

 The alerts page contains every configured and triggered alerts of all the sensors allowed to that user, with a filter to select the sensor types that are visible on the page. It has the information divided in tabs: "Configured" and "Triggered". It shows the most relevant information about each configured or triggered alert, being possible to have more information about the specific alert by clicking on the name of the alert. In the tab “Configured” it is possible to create an alert by clicking on the button “Add alert” and filling the popup form. To add an actuator, there is a button “Add actuator”. In the tab “Triggered”, all triggered alerts are displayed by date and it is also possible to delete them.


Alerts page

 There is a view for each alert. It has a section with general information about the alert and a section dedicated to the configurations, with a list of all targets associated to the alert. It also has a section with all occurrences and below a chart that displays the occurrences by day.


Mobile application

 On the initial view of the mobile application, the user can see a dashboard with seven cards, one for each type of sensor. Each card will have the values gathered from the sensor that is currently closer to the user in real-time. On the top right corner the user can select which cards are visible, or select them all.

APIs available for the sensors

 When the sensors are set up, the APIs available on the platform can be used to add the sensors. To add a sensor to the platform, a 'POST' request must be sent to /czb/devices. The body of the request must contain the following fields:

  • device_ID - Id of the device
  • device_name - Name of the device
  • provider - Name of the device provider
  • description - Description of the device
  • verticals - Verticals of the device
